# IMPLEMENTATION_TASKS.md

## Цель

Ниже приведён пошаговый технический план реализации проекта Telegram-бота для управления VPN-сервисами, конфигами клиентов и статистикой трафика.

Документ предназначен для агента-реализатора. Его задача — использовать этот план как основной roadmap и придерживаться архитектуры из `ARCHITECTURE.md`.

---

## Общие правила реализации

1. Не отходить от модели `servers -> connection -> host_actions -> providers`.
2. Не смешивать transport-слой и provider-логику.
3. Не размещать host/system actions внутри provider modules.
4. Telegram handlers не должны напрямую реализовывать бизнес-логику.
5. Все важные действия должны быть покрыты логированием и обработкой ошибок.
6. Опасные действия должны иметь явное подтверждение.
7. Код должен быть модульным и пригодным для расширения новыми провайдерами.
8. Основной UI должен быть реализован через inline-клавиатуры.
9. Любое входящее сообщение пользователя должно пересылаться администраторам, если это не часть FSM-сценария.
10. Слой статистики должен быть единым для всех провайдеров.

## Обязательные критерии качества

1. Все server/provider действия доступны только через service layer.
2. Все callback actions проходят проверку прав.
3. Все длительные операции имеют понятную обработку ошибок.
4. Все статистические выборки имеют единый формат.
5. CSV-экспорт работает стабильно и не ломает Telegram UX.
6. Пользовательский интерфейс не опирается на reply-клавиатуры как основной механизм.
7. Добавление нового провайдера не требует переписывания Telegram-ядра.
